# 第5回 コマ1 演習：単体テスト入門（Java / JUnit 5）

自販機を題材に、単体テストを「動かす → 書く」演習です。

---

## 事前準備（最初の1回だけ）

1. **親フォルダを作る**
   PC上に、授業用の親フォルダを1つ作成する。
   （3コマ分の演習は、すべてこの親フォルダの中で行います）

2. **VSCode で開く**
   VSCode の「フォルダーを開く」→ いま作った親フォルダを選ぶ。

3. **GitHub でリポジトリを作成**
   GitHub で、自分用のリポジトリを1つ作る。

4. **クローンする**
   VSCode のターミナル（`Ctrl + @`）で、次を実行する：

   ```
   git clone https://github.com/ユーザー名/リポジトリ名.git
   ```

   → 親フォルダの中に演習用フォルダができる。それを VSCode で開く。

> ※ 初回のテスト実行は、JUnit の取り寄せに少し時間がかかります。

## ファイルの場所

```
src/main/java/vending/      ← テストされる側（読むだけ・完成済み）

src/test/java/vending/      ← テストを置くところ
    CalcTest.java             演習①-2：実行して確認（すでに書いてある）
    VendingMachineTest.java   演習①-3：仕様書を見て値を入れる
```

---

## 演習①-2：用意されたテストを動かす

`CalcTest.java`（`add(2, 3)` の答えが 5 になるかのテスト）は、**すでに書いてあります**。

1. `CalcTest.java` を開く。
2. メソッド左の **緑チェック**（実行マーク）をクリックして実行 → **Green（成功）** を確認する。
   ※ ターミナル派は `mvn test` → 最後の **BUILD SUCCESS** が成功の合図。
3. 期待値の `5` を、わざと `6` に変えて実行 → **Red**（赤）になり、`expected: <6> but was: <5>` が出る。これを読む。
4. `5` に戻して、もう一度 **Green** に戻す。

ねらいは「テストが緑＝プログラムが期待どおり動いている」と「赤（失敗）の読み方」を、
自分でコードを書く前に体験することです。ここは **実行して見るだけ**です。

## 演習①-3：仕様書を、テストコードにする

`VendingMachineTest.java` を開き、**テストケース仕様書（配布プリント）** を見ながら値を入れていきます。
使う道具は `assertEquals` だけです。

- **ケース1・2（正常系）** … いくら入れるか・お釣りはいくらか、という **数字** を仕様書から入れる。
  （ケース1：150円→お釣り0 ／ ケース2：200円→お釣り50）
- **ケース5・6（境界値）** … `assertEquals(true / false, vm.canBuy("水"))` の **true / false** を仕様書どおりに直す。
  （ケース5：140円→買えない ／ ケース6：150円→買える）
- **ケース3・4（異常系）** … `assertThrows` という新しい道具が要るので、**設計まででOK**。コードにするのはコマ2で扱います。

**進め方**：1ケースずつ値を入れて、そのテストの **緑チェック** で実行 → 赤の `expected / but was` を読んで直す
→ **ケース1・2・5・6 の4本すべて Green** になればゴール。

> 答えはこのファイルには書いていません。**仕様書プリントを見て**入れてください。

---

## つまずいたら（Red / エラーのとき）

| 症状 | 原因・対処 |
|------|------------|
| `expected: <50> but was: <80>` | 仕様書の値と違う。入れた金額・お釣り、true/false を見直す |
| いつまでも Red のまま | 値がまだ仕様書どおりでない。`expected`（期待）と `but was`（実際）を見比べる |
| `mvn: command not found` | Maven が未インストール。VSCode の Java 拡張（緑チェック）で実行すればOK |
| 文字化け | ファイルを UTF-8 で開き直す |
